# ProjectComplex

Desktop-приложение для анализа дорожных изображений: обнаружение транспортных средств, оценка дистанций, извлечение EXIF/геоданных и геопривязка результатов на карту.  
Проект объединяет:
- C++/Qt GUI (интерфейс, визуализация, экспорт),
- Python ML/CV-пайплайн (YOLO, постобработка, гео-оценка),
- мост C++ -> Python через отдельный runner-процесс.

## 1. Что делает программа

`ProjectComplex` автоматизирует анализ снимков дорожной сцены:

1. Загружает изображение.
2. Обнаруживает транспортные объекты с помощью YOLO.
3. Вычисляет/оценивает расстояния до объектов.
4. Извлекает EXIF и геометаданные.
5. В полном режиме строит гео-оценку:
   - грубый подбор геопозиции (`coarse geo search`),
   - уточнение позиции/азимута камеры (`camera refine`),
   - проекцию обнаруженных ТС на карту (`vehicle geo projection`).
6. Показывает результат в интерфейсе и экспортирует в машиночитаемые форматы.

---

## 2. Основные возможности

- Запуск отдельных кластеров обработки (`cluster_1 ... cluster_4`) и полного пайплайна (`distance_full`).
- Поддержка выбора вычислительного устройства: `auto`, `gpu`, `cpu`.
- Поддержка пользовательской модели YOLO (`.pt`, `.onnx`).
- Извлечение EXIF и показ метаданных в UI.
- Карта с online/offline источниками плиток OSM.
- Ручная геопривязка (sidecar-файл `*.georef.json`).
- Экспорт результатов в `CSV`, `JSON`, `TXT`, `HTML`, `SQLite` (а также `XLSX`, если поддержка доступна в окружении).

---

## 3. Архитектура и устройство

Система состоит из двух контуров:

1. **GUI-контур (C++/Qt)**
   - Управляет сессией пользователя.
   - Отображает результаты анализа.
   - Формирует запуск Python runner.
   - Собирает stdout/stderr и итоговый `result.json`.

2. **ML/CV-контур (Python)**
   - Выполняет детекцию и постобработку.
   - Реализует кластерные и full-distance сценарии.
   - Формирует единый контракт результата (`vk_envelope_v1`).

Обмен между контурами идет через запуск внешнего Python-процесса и JSON-файлы.

## 4. Структура репозитория

Ключевые директории:

```text
ProjectComplex/
  CMakeLists.txt
  config/
    app_config.json
  src/                      # C++/Qt GUI
  resources/                # QRC/QML ресурсы
  qml/                      # вспомогательные QML
  py_src/                   # Python runner и пайплайны
    runner.py
    result_contract.py
    default_config.json
    prototypes/
    full_distance/
    notebooks/
    tools/
  .github/workflows/        # CI/CD
```

Ключевые C++ файлы:
- `src/main.cpp` - запуск приложения, splash, загрузка конфигурации.
- `src/AppConfig.*` - загрузка и нормализация `config/app_config.json`.
- `src/MainWindow.*` - компоновка вкладок.
- `src/RunnerClient.*` - запуск Python и чтение результата.
- `src/ClustersTab.*` - запуск cluster-модулей.
- `src/FullDistanceTab.*` - запуск полного расстояния.
- `src/MapTab.*` - карта, геопривязка, sidecar.
- `src/ResultView.*` - визуализация детекций, таблиц, артефактов.
- `src/ModelTypes.*` - C++-контракт структуры результата.

Ключевые Python файлы:
- `py_src/runner.py` - единая CLI-точка входа для задач.
- `py_src/result_contract.py` - сбор envelope-ответа.
- `py_src/prototypes/api.py` - общие функции обработки.
- `py_src/prototypes/cluster_1.py ... cluster_4.py` - кластерные адаптеры.
- `py_src/full_distance/full_distance.py` - полный pipeline.
- `py_src/full_distance/coarse_geo_search.py` - грубый геопоиск.
- `py_src/full_distance/camera_refine.py` - уточнение камеры.
- `py_src/full_distance/vehicle_geo_projection.py` - позиционирование ТС.

## 7. Быстрый старт

1. Установите зависимости Python:
   ```bash
   pip install -r requirements.txt
   pip install -r py_src/requirements.txt
   ```

2. Подготовьте YOLO-модель (например, положите файл в папку `yolo/`).

3. Проверьте `config/app_config.json` и `py_src/default_config.json`.

4. Соберите и запустите GUI (см. раздел ниже), либо запустите `py_src/runner.py` напрямую.

## 8. Сборка и запуск GUI

```bash
cmake -S . -B build -G Ninja
cmake --build build --parallel
```

Далее запустите бинарник `traffic` из директории сборки.

- Линкует Qt-модули.
- Копирует рядом с приложением:
  - `config/`

  - `assets/` (если присутствует)

## 9. Запуск Python runner напрямую

Базовый шаблон:

```bash
python py_src/runner.py ^
  --task cluster ^
  --module cluster_1 ^
  --input path\to\image.jpg ^
  --output-dir path\to\out ^
  --device auto ^
  --config py_src/default_config.json ^
  --result-json path\to\out\result.json
```

Для полного режима:

```bash
python py_src/runner.py ^
  --task distance_full ^
  --input path\to\image.jpg ^
  --output-dir path\to\out ^
  --device auto ^
  --config py_src/default_config.json ^
  --result-json path\to\out\result.json
```

Поддерживаемые задачи runner:
- `cluster`
- `distance_full`
- `exif`
- `preview`

## 10. Конфигурация

### `config/app_config.json`

Назначение:
- Пути к Python-интерпретатору, runner и базовому конфигу.
- Настройки вкладок и модулей.
- Map-параметры и базовые UI-опции.

### `py_src/default_config.json`

Содержит параметры пайплайна:
- detector (модель, confidence, размер входа),
- rotation/objects/cluster-параметры,
- дистанционные/геометрические коэффициенты,
- cleanup/filters.

### Runtime-конфиги

- `ui.ini` - запоминает последние пользовательские выборы.
- Временный `run_cfg.json` - собирается перед запуском runner.
- Sidecar `image.georef.json` - ручные поправки геопривязки.

## 11. Входные и выходные данные

### Вход

- Изображения:
  - `jpg`, `jpeg`, `png`, `bmp`, `webp`, `tif`, `tiff`, `heic`, `heif`
- Конфиг JSON.
- Вычислительное устройство (`auto/gpu/cpu`).
- Модель YOLO.

### Выход

Основной результат - JSON в едином формате (`vk_envelope_v1`), включающий:
- `meta` (служебная информация),
- `module` (ядро результата конкретной задачи),
- `images` (артефакты изображений),
- `tables` и `plots`,
- `exif`,
- `map`.

Дополнительные артефакты (в зависимости от режима):
- Annotated image с bbox.
- Очищенные/промежуточные изображения.
- CSV-таблицы с дистанциями.
- `coarse_geo_search.json`
- `camera_refine.json`
- `vehicle_geo.json`

## 14. Запуск через exe (пользовательский гайд)

Ниже — сценарий работы именно для пользователя готовой сборки (.exe).

### 14.1. Подготовка файлов рядом с exe

1. Распакуйте релизный архив полностью и запускайте `traffic.exe` из корневой папки распаковки.
2. Веса YOLO должны лежать **внутри папки `yolo`**. В интерфейсе есть проверка: приложение откажется использовать модель, если указанный файл не находится в ожидаемой директории `yolo`.
3. Приложение запускает Python runner отдельным процессом. Путь к Python/runner-конфигу задается в `config/app_config.json`. Если сборка portable, обычно внутри архива тоже есть нужные файлы (в частности, копия скриптов runner).

### 14.2. Вкладки интерфейса

В главном окне показываются вкладки:
- `Карта`
- `Кластеры (4)`
- `Полный режим расстояния`

Если в конфиге включена опция, может добавляться вкладка `Регрессия`.

### 14.3. Типовой сценарий: кластерная обработка (вкладка `Кластеры (4)`)

1. Откройте вкладку `Кластеры (4)`.
2. Нажмите кнопку `Выбрать фото` и выберите изображение.
3. Нажмите кнопку `Модель YOLO`:
   - выберите файл весов `*.pt` или `*.onnx`;
   - приложение импортирует/подключает модель в папку `yolo` (если это не запрещено правилами внутри проекта).
4. В выпадающем списке `Device` выберите `auto`, `gpu` или `cpu`.
5. Нажмите кнопку нужного кластера (кнопки вида `1: <название>`, `2: <название>` и т.д.).
6. Результат и логи отображаются в блоке `ResultView`:
   - `Консоль` — текстовые сообщения и прогресс,
   - `Таблицы` — вкладки `Detections` и `EXIF`,
   - `Графики` — дополнительные визуализации (если есть).

После завершения расчета вкладка `Карта` обновится автоматически (если вы используете связку вкладок `imageSelected/resultReady`).

### 14.4. Типовой сценарий: полный режим расстояния (вкладка `Полный режим расстояния`)

1. Откройте вкладку `Полный режим расстояния`.
2. Нажмите `Выбрать фото`.
3. Укажите YOLO модель через `Модель YOLO` (веса должны быть в `yolo`).
4. Выберите `Device`.
5. Нажмите кнопку `Полный режим`.
6. Далее смотрите результат в `ResultView` так же, как в кластерной вкладке.

### 14.5. Работа с картой и ручная геопривязка (вкладка `Карта`)

Если для вашей сцены точность недостаточна, используйте ручную корректировку.

1. Перейдите на вкладку `Карта`.
2. Нажмите `Точка камеры` и щелкните по карте, чтобы задать положение камеры (latitude/longitude).
3. Нажмите `Направление` и щелкните по карте второй точкой, чтобы задать азимут камеры.
4. При необходимости используйте `Сброс`.
5. Нажмите `Сохранить`, чтобы обновить данные геопривязки.

Ручные значения сохраняются рядом с исходным изображением в sidecar-файл:
- `<image.ext>.georef.json`

Sidecar включает координаты камеры и азимут (а также служебное поле `location_source`, uncertainty и т.п.).

Если приложение использует EXIF как подсказку, оно показывает текстовые подсказки (например, что координаты взяты из EXIF и может быть рекомендована ручная проверка).

### 14.6. Сохранение результатов и экспорт

Внутри вкладки с результатом (обычно `ResultView`) доступны кнопки:
- `Сохранить изображение...` — сохранить визуализированный результат,
- `Экспорт данных...` — сохранить данные в файл.

При экспорте выбирайте формат из списка:
`CSV`, `JSON`, `XLSX`, `SQLite DB`, `TXT`, `HTML`.

### 14.7. Где искать временные файлы

Во время работы приложение создаёт рабочие директории в системной временной папке в подпапке `traffic_work`:
- например, `traffic_work/full_distance/...` или `traffic_work/cluster_<id>/...`

Во время выполнения `result.json` находится внутри соответствующей временной директории. После завершения приложение обычно очищает рабочую папку `traffic_work`, поэтому для отладки может потребоваться экспорт или копирование артефактов во время работы.

Для карты также используется временный файл QML `temp_map.qml`, который записывается рядом с `traffic.exe`. Убедитесь, что у системы есть права на запись в папку с приложением.

## 16. Ограничения и известные нюансы

- Веса моделей (`.pt`) не хранятся в репозитории и должны поставляться отдельно.
- Для части notebook-модулей могут потребоваться дополнительные Python-пакеты (`pandas`, `matplotlib`).
- Полный гео-режим зависит от качества EXIF/GPS и параметров сцены.
- На Windows корректная раскладка Qt runtime обычно требует `windeployqt`.
- Для offline-карт должны быть доступны локальные тайлы/кэш в ожидаемом формате.

## 17. Диагностика и troubleshooting

### 1) Python runner не стартует

Проверьте:
- путь к Python в `app_config.json`,
- наличие `py_src/runner.py`,
- установлен ли набор Python-зависимостей.

### 2) Ошибка загрузки модели

Проверьте:
- путь к файлу модели,
- права доступа к файлу,
- совместимость модели с вашей версией `ultralytics/torch`.

### 3) Пустые детекции

Причины:
- слишком высокий confidence threshold,
- неподходящая модель,
- сложные условия кадра (освещение/ракурс/разрешение).

Что сделать:
- снизить порог confidence,
- попробовать другую модель,
- проверить исходное изображение.

### 4) Карта не отображается или без гео-результата

Проверьте:
- наличие EXIF/GPS,
- доступ к online тайлам или корректность offline-источника,
- корректность входных артефактов full-distance модуля.

### 5) Проблемы с кодировкой/логами Python

Приложение запускает Python с `PYTHONUTF8=1` и `PYTHONIOENCODING=utf-8`; если в окружении есть переопределения, уберите конфликтующие переменные.

## 18. Для разработчиков

### Рекомендации по расширению

- Новый режим обработки добавляйте как новую задачу в `py_src/runner.py`.
- Контракт результата держите совместимым с `py_src/result_contract.py` и C++-типами в `src/ModelTypes.*`.
- Для новой вкладки GUI:
  - создайте `*Tab` в `src/`,
  - подключите в `MainWindow`,
  - используйте `RunnerClient` для запуска Python-задач.

### Чеклист после изменений

1. Сборка C++ проходит.
2. Runner запускается вручную на тестовом изображении.
3. JSON-контракт валиден и читается GUI.
4. Артефакты открываются в `ResultView`.
5. Экспорт работает хотя бы в `CSV` и `JSON`.
